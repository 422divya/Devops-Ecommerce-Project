==> ❌ AWS credentials in tfvars are NOT used for backend authentication

Terraform backend (S3 + DynamoDB) needs AWS credentials from:

✔ Environment variables
✔ AWS CLI profile
✔ IAM role (EC2 only)

NOT from:

terraform.tfvars
.tf variables
input variables

==> The warning message above is now we can use the DynamoDB based locking.  It now prefers S3 native locking using a lock file

use_lockfile = true

Above uses Amazon S3 built-in locking mechanism. No separate DynamoDB table needed.


Before:
S3 → stores state
DynamoDB → handles locking
Now:
S3 → stores state + handles locking

4- Frontend webpage was loading but backed was not loading. It was giving error "Error: TypeError: Failed to fetch"

5- Because your browser is outside the cluster, it cannot resolve or connect to the internal service name backend.

Your frontend runs in the browser, not inside Kubernetes.

So this line in your frontend:

fetch('http://backend:8080')

is executed on your laptop/browser, not in the cluster                                                                




6- What changed:
Added frontend-nginx-config ConfigMap with nginx.conf
Frontend now proxies browser requests from /api to http://backend:8080
Frontend still serves the static HTML page from frontend-content



7- Flow:

Browser calls the frontend LoadBalancer URL.
LB forwards to the frontend Service.
frontend Service routes to a frontend pod.
Frontend nginx receives the browser request.
For /api, nginx proxies internally to http://backend:8080.
The backend Service routes that to a backend pod.
Backend pod responds, and nginx returns the response to the browser.
So the browser only talks to the frontend LB. The backend is reached from inside the cluster via nginx proxy in the frontend pod.
